# 小鸡AI App 完整架构提炼 × AICrew 映射蓝图

> 任务：小鸡AI 是一个 App，把它的**完整架构提炼出来**，给一个完整解决方案。
> 此前 4 轮 sprint 是**逐平台碎片化**映射（小红书/抖音/视频号各研究一遍）；本文档**跳出碎片化**，对小鸡AI 做一次**整体逆向架构** + **全平台×全模块映射矩阵**，把"还差什么"系统性摆上台面。
> 交付物形态：**仅架构提炼蓝图（零代码改动，L0）**。落地哪些缺口由本文档第 7 节交回用户拍板。

## 0. 一句话结论

小鸡AI 不是"一个 App"，是 **一个品牌下的两条产品线**：① **小鸡AI App**（移动端，官网露出，6 个内容创作模块，合规友好）；② **小鸡AI 桌面矩阵系统 V6.x / 红薯矩阵**（指纹浏览器 + RPA，10 个自动化模块，风控对抗区）。
**完整架构 = 内容创作层（App）+ RPA 自动化层（矩阵系统）两层叠加**。AICrew 的完整解决方案是：把 App 的**全部 6 个内容模块**在**全部已接平台**上对齐（这是可量化的有限缺口），并把 RPA 自动化层按**黑/灰/绿边界谱系**精确处理（🔴黑区永久焊死、🟡灰区有条件、🟢绿区可合规承接，详见第 6 节深拆）。本文档把这两件事都拆成可勾选的矩阵。

## 1. 提炼方法论（"如何把一个 App 的架构提炼出来"）

提炼一个**外部闭源 App** 的架构，不是猜，是 4 步可复用流程（本次即按此执行，可固化为以后拆任何竞品的标准动作）：

1. **双线分离**：先判定产品有几条形态不同的产品线（移动 App / 桌面客户端 / 浏览器插件 / SaaS 后台）。形态决定能力边界——把不同形态混为一谈是架构提炼的头号错误（小鸡AI App ≠ 矩阵系统）。
2. **模块逐条落表 + 标来源置信度**：每个模块必须有出处（官网 / 官方文档 / 第三方测评），标 `sourced`（逐字）/ `inferred`（推断）。**外部 App 的能力声明绝不杜撰**——这是 fidelity 红线，等同"没拿到设计 screenshot 不能宣称像素级"。
3. **每模块定性分层**：把每个模块归到 🟢内容生成层 / 🔴RPA 自动化层 / 🟡数据抓取层。这一步决定本项目能不能/应不应"集合进来"。
4. **对照本项目逐格映射**：本项目侧的每一格都**读代码核实**（ground truth），不靠记忆。产出 `已覆盖 / 净缺口 / 边界外` 三态矩阵。

> 复用法则（承接已确立的"映射法则"）：**小鸡AI 每个能力 → 取其「内容产物」半降维进 AICrew**；自动化执行那一半留在边界外，由人工执行。

```

        ┌───────────────────┴───────────────────┐

   M1 文案生成                         X1 采集 / X2 去重
   M2 图片配文                         X5 多账号 / X6 养号 / X9 指纹浏览器

        │                                       │
   ✅ AICrew 域（可做，本文档第 4 节映射）   🔴🟡🟢 边界谱系（第 6 节深拆）
```

**架构洞察**：M1–M6（App 内容层）与 X3/X4（矩阵里的内容生成那一半）**完全重叠**——矩阵系统的"内容生成"就是 App 内容能力 + RPA 自动化封装。所以 **AICrew 要对齐的"完整架构"= App 的 6 个内容模块**，把它们在全平台铺满即达成内容层 parity；矩阵系统只是在这 6 个能力外面套了一层自动化执行壳，那层壳是边界外。

### 4.1 矩阵读出的 3 类系统性缺口

1. **横向不齐（同模块只在小红书落地，其他平台缺）**：
   - **M2 图片配文**：仅小红书有 → **抖音/视频号/通用 各缺 1**。
   - **M4 发布管理**：仅小红书有 publish_optimizer + 交接 Tier0 → **抖音/视频号 各缺 1**（发布优化清单 + 官方交接）。
2. **纵向结构不对称**：**视频号漏斗缺 measurement(复盘) 阶段**（rednote/douyin 都有，channels 只有 8 阶段）。这违反"每平台漏斗结构同构"的隐性不变量，导致 M5 数据分析在视频号侧无落点。
3. **半成品散落（有能力无产物）**：**M3 热点话题**靠通用 `trend` agent，但只有小红书把它沉成独立选题技能（topic_mining/calendar）；抖音/视频号的"热点选题"散在 content stage 技能里，无独立"热点话题推荐"技能——用户在选择器里搜不到对应卡片。

## 6. RPA 自动化层重点考察（完整提炼 + 黑/灰/绿边界谱系）

> 本节【替换并深化】原第 6 节单薄的「双重不可逾越」三段焊死表述。原版只说「RPA 层永久焊死在边界外」，把整层当作一块铁板禁区挥手略过——这对一次声称「完整架构提炼」的蓝图是结构性残缺：自动化半壁才是小鸡AI 矩阵系统的招牌与命门，不认真拆它，就既看不清边界为何不可逾越，也错失了边界内侧真实可合规承接的价值子集。本节把 RPA 层重构为一条 **黑 🔴 → 灰 🟡 → 绿 🟢 的边界谱系**：精确论证「为什么不可逾越」（机制级技术 + 条款级合规），并标出「哪一小块可以、在什么条件下、用哪些已有基建合规承接」。所有外部 App / 平台能力均标注 `sourced`（出处逐字）/ `inferred`（推断）/ `derived`（由前两者推导），fidelity 红线不杜撰。
>
> **本次修订要点（对抗审查后纠偏）**：绿区/黑区的唯一分界是「逐帖人工确认 gate 是否保留 + 人是否在环」，**不是「是否走官方授权通道」**。据此把「抖音服务端 upload-video + video-create 单号发布」从绿区整体移入灰区——它一次 OAuth 授权后即由服务器持 access_token 机器代发、无逐帖用户点发布动作，且必须打破静态站「无后端进程」不变量，完全命中本草稿自定义的灰区判据。同时区分「平台审核」与「人工确认 gate」：抖音 create 后的「审核期仅自己可见」是平台侧内容审核，**不是 user-in-loop 确认**，不能拿来给绿区贴标签。

### 6.0 为什么 RPA 层必须被认真提炼（而非挥手略过）

完整架构提炼的第一性原则是「双线分离 + 不漏半壁」。小鸡AI = 内容创作层（App，6 模块）+ RPA 自动化层（矩阵系统 V6.14，10 模块）两层叠加；矩阵系统的招牌「一键克隆」、桌面指纹浏览器、多账号铺量，全部活在 RPA 层。把这层当作「一句话焊死」处理，会犯三个错误：

1. **看不清边界的真实形状**：以为「RPA = 一整块禁区」，实则它是 `采集🔴 → 去重🔴 → 改写🟢/生成🟢 → 多账号🔴 → 养号🔴 → 发布🔴 → 互动🔴 → 数据🟡` 的一条流水线，每个环节的层属性不同——「一键克隆」内部就同时含 🔴采集 和 🟢二创改写。一刀切会把 🟢内容产物半也误判为禁区，丢失差异化机会。
2. **错失边界内侧的合规子集**：X7 自动发布里「机器代操作账号批量群发」那半是 🔴，「单号经官方开放平台 OAuth 授权的服务端发布」那半是 🟡（官方授权但仍是机器代发、无逐帖人工 gate），而「内容打包一键带稿交接」「内容排期到点提醒」这几半才是 🟢——它们是 App「一键发布」丝滑感的**真实底层**，本就不是 RPA。
3. **论证不到位则边界会漂移**：只说「做不到 + 不应做」而不给机制级证明，未来某次需求评审就可能动摇。本节用同源策略 / 浏览器沙箱 / 进程模型逐能力论证「物理做不到」，用 2025 各平台具体条款论证「合规不应做」，把焊死钉死在第一性原理上。

> 提炼法则（承接已确立的「半降维映射」）：小鸡AI 每个 RPA 能力 → 取其「内容产物 / 官方交接（人在环）」半合规降维进 AICrew；「机器代操作账号」那半永久留在边界外。注意：**「官方授权的机器代发」既不属内容产物半、也不属人在环交接半——它是落进灰区的第三类**，本节明确把它与绿区切开。

#### 6.1.2 闭环协同架构总结（不是 10 个并列功能，是一条流水线 + 一个底座 + 一个回路）

小鸡AI 矩阵系统的 RPA 层是一个 **以指纹浏览器为底座、数据驱动自循环的有机闭环**：

```
        ┌──────────────────── 闭环主链（sourced 模块名 + derived 串接）────────────────────┐
进料        洗稿（招牌「一键克隆」=X1+X2+X3+X4）        分发底盘    预热      执行           回收
X1采集🔴 → X2去重🔴 + X3改写🟢 + X4生成🟢   →   X5多账号🔴 → X6养号🔴 → X7发布🔴 + X8互动🔴 → X10数据🟡
  │                                                                                            │
  └──────────────── 回灌选题/发布策略，形成数据驱动优化循环 ◀──────────────────────────────────┘

        ┌──────────── 使「批量而不被封」的三层底座（sourced/inferred）────────────┐
        X9 指纹浏览器🔴（每号独立设备指纹，sourced 底座）
      + I2 代理 IP 池🔴（独立网络出口，账号-指纹-IP 三元组，inferred）
      + I1 任务调度引擎🔴（多线程编排 + 定时打散，inferred）
      + I3 拟人化行为引擎🔴（机器节奏→人类节奏，derived）
```

**命门（`derived`）**：指纹 + IP 双隔离让 N 个账号在平台看来是 N 台独立真实设备，养号/发布/互动才得以批量运行而不被关联封号——这是整个 RPA 体系赖以存在的前提。抽掉 X9/I2，矩阵立刻坍缩为「一个账号被批量异常行为标记」。

**关键架构分水岭（`sourced`）**：同样叫「发布」，矩阵线是真·机器代操作账号（浏览器自动化模拟点击 / 逆向接口），而 App 线是官方分享 SDK + URL Scheme 深链交接、用户手动确认——两条产品线在「发布」这一环彻底分叉。这条分叉正是 6.4 绿区承接的入口；而**官方授权下的服务端机器代发**（如抖音 video-create）虽合规，但同样删掉了「用户手动确认」这一动作，故归 6.5 灰区而非绿区。

### 6.2 双重不可逾越的【精确】论证

「焊死在边界外」不是态度，是两条第一性原理。下面逐能力做机制级证明。

#### 6.2.1 技术边界：浏览器同源策略 / 沙箱 / 进程模型逐能力论证（`derived`）

AICrew 运行时事实（已核实代码，`verified`）：`AICREW_STATIC_EXPORT=1 → output:"export"`（`next.config.mjs`）编译为纯静态文件，**应用没有自己的常驻进程**；`generateText/Image/Video`（`lib/ai/providers.js`）用全局 fetch 把 token 放进**浏览器**请求头直连 Claude/OpenAI（或经 `provider:"system"` 的无状态 `/api/ai/generate` 代理转发），中间无任何自动化执行层；唯一「服务端」是请求作用域的无状态 /api（`routeGuard` 限流用进程内存 Map、随实例重置，**无持久队列/cron/worker**）；`lib/db/client.js` postgres 直连「绝不进客户端 bundle」，只连自家 Supabase，**从不连 xhs/douyin/wechat**；代码里**不存在任何外部控制通道**（无 headless driver / CDP socket / child_process / 扩展 host_permissions / OS 无障碍接入点）。

| RPA 能力 | 被哪个浏览器机制拦死 | 机制级论证 |
|------|------|------|
| **X1 采集 / 一键克隆采集半** | 同源策略（SOP）网络读取边界 | 对 xhs/douyin 跨源 fetch：cors 模式因目标不回 ACAO、preflight 失败而 reject；no-cors 返回 opaque response（status 0、body 对 JS 不可读）。两条路都拿不到字节。SOP 是渲染进程安全边界，页面 JS 关不掉——只有带 host_permissions 的扩展或自带网络栈的桌面客户端能绕，静态站两者皆无 |
| **X5 多账号管理** | 浏览器引擎/OS/网络层身份隔离 | 多账号反检测要求每号独立 cookie 分区 + 存储隔离 + 独立代理 IP + 独立设备指纹。单页只有一个 cookie/storage 分区、一个源 IP，无法为 N 号各开隔离 BrowserContext。要实现就得「成为浏览器」，而非运行在浏览器之内 |
| **X6 智能养号** | 进程模型 + 后台执行节流 | 养号需 per-account 常驻并行长跑执行线（本质 OS 进程）。浏览器只有前台 tab 单事件循环；后台 tab 定时器被 clamp 最终冻结/丢弃（bfcache/tab discard）；无人开页时静态站无任何自有进程可运行。Service Worker / Periodic Background Sync 被重度限流（需已装 PWA + 用户参与度 + 最小间隔约 12h），且仍无法驱动跨源已登录写操作 |
| **X7 自动发布** | 无常驻调度器 + 跨源 credentialed 写被拦 | 「定时」需计划时刻自唤醒的常驻调度器，静态站只是一堆文件，无自有 cron；唯一会跑代码的 /api 请求到来即跑、跑完即灭。「批量跨号」还叠加 X5/X9 多账号隔离不可得。即便有定时，跨源向 xhs 发布 API 写入还需签名参数（x-s/x-t）+ 用户平台 session，cross-origin credentialed 被 CORS 拦、cookie 不随发 → 物理写不出去 |
| **X8 自动互动** | DOM 层 SOP + isTrusted + 凭证式 CORS（沙箱三拦） | (1) 跨源访问 xhs.com 的 window/frame DOM 抛 SecurityError；(2) 页面合成输入事件 isTrusted=false，平台写操作拒绝非可信事件，且无法向跨源文档派发事件；(3) 直调写 API 需签名 + 平台 session，跨源 credentialed 被 CORS 拦、SameSite cookie 不随发。浏览器沙箱存在本意正是阻止一个站点冒充用户在另一站点行动（CSRF/clickjacking 防护）。RPA 靠站在沙箱之外（OS 无障碍/合成输入/CDP）绕过，页面 JS 没有这些通道 |
| **X9 指纹浏览器** | 指纹由宿主浏览器/OS/网络设定，对页面只读 | navigator.userAgent、screen、WebGL renderer、Canvas 哈希、TLS/JA3 指纹、源 IP 全由宿主设定，页面只读，无法伪造；也无法派生 N 个各自隔离 cookie 分区 + 独立代理的上下文。X9 本质是浏览器引擎/OS 能力，不是引擎里的页面能力 |
| **X10 实时数据抓取** | 跨源不可读 + 无常驻轮询面（双重落空） | 已登录数据端点跨源：无 CORS 头则 reject 或 opaque 不可读；读私域数据需平台 session cookie，跨源不随发（SameSite + 凭证式 CORS 双拦）。即便单次拿到，「实时」还要常驻轮询调度器（与 X7 同样不可得）→ 既读不到字节、也没有常驻拉取面 |

> 一句话技术结论：要做这些，必须「成为浏览器 / 站在沙箱之外 / 拥有自有进程」三者之一——静态站三者皆无。这不是工程量问题，是软件形态问题。

#### 6.2.2 合规边界：2025 各平台具体红线条款（`sourced`）

即便技术上能做，下列条款也禁止——这是「不应做」的独立第二重。

| 平台 | 条款 | 直接覆盖的 RPA 能力 |
|------|------|------|
| 小红书《社区公约2.0》（2026-01 上线）/社区规范 | 禁止恶意批量注册、批量发布、高频发布、机器发布等非正常模式发布笔记 | X5 / X7 |
| 小红书 | 禁止直接或变相诱导关注/点赞/评论/收藏；2025 上半年封禁黑灰产账号超 **1200 万**、处置虚假营销笔记 1376 万、清理虚假评论超 3.6 亿，养号涨粉+机器互动为治理重点 | X6 / X8 |
| 小红书 | 禁止搬运其他平台图文/视频，处罚=删除+扣信誉分，累计 **3 次限流、5 次以上封号** | X1 / X2 |
| 抖音《用户服务协议》 | 不得使用任何插件/外挂/第三方工具，包括任何自动化程序/软件接入抖音、收集或处理其中信息内容 | X1 / X5 / X7 / X8 / X9 |
| 抖音《社区自律公约》 | 禁止买卖账号、养号等作弊违规；官方 API 发布视频须经平台审核（审核期仅自己可见）、禁品牌水印（否则降权/下架） | X6 + 框定 6.5 灰区官方 API 边界 |
| 微信《视频号运营规范》 | 未经许可使用插件/外挂/第三方工具接入微信及相关系统属违规（覆盖多开/群控/协议号） | X5 / X9 |
| 微信《视频号运营规范》 | 禁止利诱或胁迫用户分享/关注/点赞/评论 | X8 |
| 微信《视频号运营规范》 | 禁止利用第三方平台/外挂/系统漏洞刷粉/赞/评论/阅读量；处罚含禁转发、删文、删粉、封禁能力直至封号 | X6 / X8 / X10 |

> 合规结论：指纹浏览器 X9「存在目的即对抗平台风控」，被微信/抖音外挂条款直接覆盖；养号/批量发布/自动互动/搬运去重逐条命中具体罚则。AICrew 既定边界——**只产内容/话术/脚本/SOP/策略，把合规风险留在「人来执行」**——正是差异化护城河。严禁把灰区/黑区粉饰成绿区：合规越界比保守更糟。**特别地：「官方授权」只洗白合规维度，不改变「机器代操作账号、无逐帖人在环」这一行为本质，故官方授权的服务端代发归灰区不归绿区。**

### 6.3 黑 / 灰 / 绿三段边界谱系（把「单块禁区」重构为谱系）

原版把 RPA 当作一块铁板 🔴。真实形状是一条沿「人是否在环 / 是否走官方授权通道 / 是否对抗风控」三条轴展开的谱系。**三轴并非同权——「人是否在环（逐帖人工确认 gate）」是判定绿区的决定轴，「官方授权」只决定灰区与黑区的分界，不能把能力抬进绿区**：

```
🔴 黑区（永久焊死）        🟡 灰区（有条件可做+残余风险）            🟢 绿区（可合规承接）
机器代操作账号/对抗风控  ──  官方授权的机器代发·单号·需突破静态站  ──  内容产物 / 官方交接 / 人始终在环
X1采集 X2去重 X5多账号       X7抖音 upload-video+video-create        X3改写 X4生成 一键克隆可降维半
X6养号 X8互动 X9指纹         （服务端代发，无逐帖人工确认）            Tier0/0.5 发布交接 投后复盘内容
X7批量跨号群发（RPA）半       X10官方API读自有数据                    矩阵人设/养号内容计划半
X10实时抓他人/全站数据        小红书Tier1真预填 微信小店电商API        抖音「分享到抖音」compose 交接(inferred)
```

**判定一个 RPA 能力落哪段的三个问题（按优先级，自上而下短路）**：① 是否需要机器代替用户操作账号、且对抗平台风控（指纹/逆向/无授权）？是→🔴。② 机器代操作账号但走平台官方开放平台授权通道、仅限单号/自有数据、**无逐帖人工确认 gate**？是→🟡（需突破静态站加服务端；典型即抖音 video-create）。③ 是否纯产内容产物 / 官方分享交接、**且每一帖都保留不可移除的用户手动确认动作**？是→🟢。

> 注意第②③问的分界：抖音服务端 video-create 满足「官方授权」却**不满足「逐帖用户手动点发布」**——它一次授权后服务器持 access_token 自动发，发后由平台做内容审核（platform review，非用户确认）。这道「平台审核」绝不能与「人工确认 gate」混为一谈：前者是平台事后把关，后者是用户每帖在环主动确认。故 video-create 落 🟡 不落 🟢。

### 6.4 绿区 🟢：AICrew 可【合规承接】的子集（落地路径 + 已有基建）

绿区是「半降维法则」的落点：取每个 RPA 能力的内容产物半 / 官方交接半（**人始终在环、每帖保留手动确认动作**），弃机器操作半。**绿区不含任何服务端机器代发**——凡删掉逐帖用户确认动作的能力一律下沉灰区。

#### 6.4.1 绿区能力清单

| 🟢 绿区项 | 合规机制 | AICrew 已有基建 / 落地路径 |
|------|------|------|
| X3 文案改写（内容产物） | 纯 AI 文本生成，零账号操作，人工审核后自行发布 | 已强覆盖：copy agent + `buildCopyPrompt` 经 platformPresets DNA 全平台落地，无需改架构 |
| X4 内容生成（内容产物） | 浏览器客户端 token（或 system provider 服务端代理）直连 LLM 生成创作物，不碰账号 | 已覆盖：加 skill 即经 `PROMOTION_REGISTRY` 自动进 funnel/UI，零新增架构 |
| 招牌「一键克隆」可降维半 | 用户人工提供参考素材 → AI 改写文案+生成全新图片，**绝不自动爬取**，规避「搬运」红线 | 复用 copy engine + 图片生成管线，新增一个「参考改写」内容 skill 入 registry |
| 发布交接 Tier0（WebShare/Clipboard/深链） | 官方分享交接=平台授权合规通道，非 RPA，终点恒为用户手动点发布 | **已落地**（仅小红书）`lib/share/rednote.js` 纯逻辑层；抖音/视频号镜像见 6.4.3，确定性不同 |
| 投后复盘 / 发布前自检（内容半，弃实时抓数） | 不抓平台数据，基于用户手填或官方授权导出数据，产归因复盘+优化清单 | `rednote_performance_review_v1`(domain.js:645) / `rednote_publish_optimizer_v1`(domain.js:1082) / `douyin_campaign_review_v1`(domain.js:1352) 已部分落地（小红书/抖音），补齐视频号 |
| 多账号「人设矩阵/养号内容计划」内容半（弃自动养号执行） | 只产差异化人设定位 + 内容日历 + 首评话术等纯文本，实际运营由人执行 | 新增「矩阵人设规划」内容 skill，与 X5/X6 的 RPA 执行壳完全脱钩 |

> 技能 id 一律带平台前缀（`rednote_*` / `douyin_*`），**不存在无前缀的通用 `performance_review_v1` / `publish_optimizer_v1`**；引用务必带前缀以免误指通用技能。

#### 6.4.2 官方分享交接（绿区的「人在环」入口，client-side handoff）

App「一键发布」丝滑感的真实底层不是 RPA，是**官方分享开放平台 + URL Scheme 深链交接**——其要害是「内容预填到官方发布器、由用户手动点发布」，人始终在环。AICrew 静态站可直接复用同一套机制：浏览器对 App 三类系统能力全有对等 API（系统分享面板↔Web Share、剪贴板↔Clipboard、深链↔URL Scheme、本地文件↔Blob 下载），外加官方 JS SDK 本就跨端。**本小节只收 client-side handoff（人在环）；服务端机器代发（抖音 video-create）已移入 6.5 灰区。**

| 平台 | 官方交接接口 | 能力（均保留逐帖用户确认） | 接入要求 | 置信度 |
|------|------|------|------|:----:|
| 小红书 | 分享 SDK（agora，含 JS `xhs.share`）+ `xhsdiscover://post_note/` 深链 | 一键把图文/视频预填到官方发布器，用户手动点发布（非服务端代发） | Tier0 深链零凭证；Tier1 真预填需开放平台注册应用拿 app key/secret + 服务端签名 + 域名白名单审核 | `sourced`（深链已实现） |
| 小红书 | 笔记详情 API / 评论 API（读，无写） | 读取笔记详情、读取/管理评论（无内容创建写能力） | 企业资质审核 + 接口二次审核 + App Key/Secret/access_token；个人主体受限 | `sourced` |
| 抖音 | 「分享到抖音」官方 SDK / `snssdk1128://`·`aweme://` 类 compose 深链 | 设想把视频/图文交接到抖音发布器、由用户手动确认（**对应小红书 share SDK，是抖音真正的绿区路径**） | 客户端 handoff，可能需官方分享 SDK 接入；是否支持「外部网页→抖音发布器内容预填」**待证实** | `inferred`（compose 预填能力未经证实，不与小红书深链同等确定） |
| 微信视频号 | 微信 JS-SDK 分享 / 「分享到朋友圈·会话」 | 仅能把**链接**分享到会话/朋友圈，**无「网页→视频号发布器 compose/预填」公开通道** | 微信公众平台 JS-SDK 配置 + 域名校验 | `inferred`（视频号侧 handoff 通道存疑，见 6.4.3 G6 警示） |

> 准入数字提醒（`inferred`，待逐字出处复核）：抖音开放平台企业应用认证费、各类审核工作日（如 1-3 / 1-5 日）等具体数字在不同时期、不同渠道口径不一，**凡无逐字一手出处的一律标 `inferred`**，落地前以开放平台官方文档当期口径为准，不在蓝图里当 `sourced` 硬数字使用。

#### 6.4.3 Tier 交接模型（绿区核心，把 X7 沿人工 gate 一刀切开）

| Tier | 机制 | 自动化级别 | 人在环位置 | 现状 |
|------|------|------|------|------|
| **Tier 0** 一键带稿交接（🟢） | `lib/share/rednote.js`：`buildRednoteShareText`(caption+normalizeHashtags→可粘贴 text) + `REDNOTE_PUBLISH_DEEPLINK`(xhsdiscover://post_note/) + `REDNOTE_PUBLISH_STEPS` + `supportsRednoteHandoff`（仅小红书产物开闸）；组件层做 navigator.share/clipboard/深链/Blob 下载副作用 | 准全自动「内容打包+一键唤起」，零账号自动操作；承接 X7「省去复制粘贴一键带稿」半 | 恒为用户在官方发布器手动点发布，不可绕过 | **已实现（仅小红书，`sourced`+`verified`）**；抖音/视频号镜像见下方 G6 警示，确定性显著低于小红书 |
| **Tier 0.5** 排期队列+到点提醒一键确认（🟢半自动） | 复用 Supabase history 把产物排进发布日历 + 叠 `rednote_publish_optimizer_v1`(domain.js:1082) 做就绪门 + Web Notification 到点提醒，点开落进 Tier0。调度器本质是「到点提醒」非「服务器定时执行」，靠浏览器在线触发 | 半自动：排期+提醒自动化，发布动作 100% 人工 | 双重 in-loop：①到点点「确认发布」②官方发布器手动点发布 | **可做**（基建齐备：Supabase + publish_optimizer + Tier0 三件套已在，缺排期 UI+通知接线，零新增架构零凭证零合规风险） |
| **Tier 1** 官方分享 API 真预填（🟢/需凭证） | 接小红书 `xhs.share` JS SDK：新增 `/api/xhs-share-sign` 用 app secret 生成 signature（secret 走 server env 不入客户端，AICrew /api 已具承接条件）→ 发布页内容自动预填 | 全自动「内容预填」，发布动作仍走官方交接、用户手动确认（App「一键发布」丝滑感真正来源，但是官方分享开放平台非 RPA） | 用户在发布页手动确认——官方强制 gate，第三方不可绕过（正是它与 RPA 的分界） | 需平台资质（注册应用拿 app key/secret + 域名审核），阻塞在凭证申请；AICrew /api 路由已具条件，待凭证排期 |

> **统一不变量（证明 Tier 模型是「正解」而非妥协）**：每一档只自动化「内容准备/打包/预填/到点提醒」这半边，从不自动化「在官方发布器里点发布」那半边——**官方发布器的逐帖人工确认 gate 在每一档都不可移除**。这正是 Tier 模型与抖音 video-create 的根本分野：video-create 删掉了这道 gate 改由服务器代发（故归灰区），Tier 模型沿这道 gate 把 X7 一刀切开，承接「省去复制粘贴/到点不忘发/批量准备」全部内容价值，把「代用户操作账号」永久留在边界外由人执行。这就是「发布最后一公里」既补齐又不踩红线的护城河。
>
> **⚠ G6 镜像警示（修订重点，降级为 `inferred`）**：Tier0 在小红书是 `sourced`+已实现（`xhsdiscover://post_note/` compose 深链已验证可唤起官方发布器）。但**镜像到抖音/视频号不是「纯前端零后端零凭证即可」的等价操作**：
> - **抖音侧**：`snssdk1128://`/`aweme://` 类 compose 深链能否从外部网页唤起发布器并**预填内容**未经证实，标 `inferred`，不得与小红书深链平级断言；
> - **视频号侧**：**微信无公开的「网页→视频号发布器 compose/预填」通道**——微信 JS-SDK 分享只把链接分享到会话/朋友圈，并非把素材预填进视频号发布器。故 **G6 在视频号侧能否成立存疑**，落地前必须先验证通道是否存在，不能假定与小红书 Tier0 同等确定。

### 6.5 灰区 🟡：有条件可做、有残余风险（条件 + 风险）

灰区共性：技术上需**突破当前「纯静态站无后端进程」边界新增服务端**（OAuth 回调 / access_token 托管 / 签名端点），合规上需**官方授权 + 仅限单号/自有数据**，行为上**机器代操作账号、无逐帖用户确认 gate**（这正是它不进绿区的根本原因）。任何一项做成「高频/跨号/抓他人数据」即重新跌回 🔴 黑区。

| 🟡 灰区项 | 条件 | 残余风险 |
|------|------|------|
| **X7 合规子集：单号·官方授权的服务端机器代发（抖音 upload-video + video-create）** ——【本次修订从绿区移入】 | 接抖音开放平台企业应用 + 应用单独申请并审核「视频发布及管理」能力（`inferred`，**非企业认证后默认开通、对一般开发者受限**）+ 用户 OAuth 一次性授权拿 access_token；AICrew 须新增服务端 OAuth 回调/token 托管/签名端点 | **行为本质=机器代发、无逐帖人工确认 gate**（create 用存储 token 由服务器直接发起；create 后「审核期仅自己可见」是平台侧内容审核 platform review，**不是 user-in-loop 确认**）；仍受平台审核、token 会过期、仅限单号非批量；做成高频/跨号定时即重踩「机器批量发布」红线；新增服务端=扩大攻击面与运维 |
| X10 合规子集：官方数据 API 读「自有账号」做投后复盘（抖音视频数据 / 小红书笔记详情·评论读取） | 企业资质审核 + 用户授权 + 仅读授权范围内自有账号数据 + secret server-only；配合 AICrew 既有 Supabase server-only 直连 | 第三方爬虫式「实时抓他人/全站数据」仍是黑区；官方读接口有频控+二次审核；需突破静态站加服务端；数据用途越界即违规 |
| 小红书 Tier1 真预填（xhs.share 内容预填官方发布器**由用户确认**） | 开放平台注册应用拿 app key/secret + 服务端签名端点（secret 走 server env）+ 域名/回调白名单审核 | 凭证审核可能不批个人/小主体；本质仍是 handoff 非代发（**保留人工 gate，严格说偏绿，列此处仅因需凭证+服务端**）；引入 server secret 增运维与泄露面，相比已落地 Tier0 收益边际 |
| 微信小店电商 API 承接（商品/橱窗/订单，非视频发布） | 开通微信小店 + 微信开放平台权限集授权；仅限电商履约 | 官方仅开放电商接口、**视频内容发布接口未开放**；与「内容创作」定位弱相关，承接价值有限，不能替代视频发布 |

> 灰区开放平台空白处（`inferred`，fidelity 标注）：小红书「服务端代发布/定时/批量发笔记」写接口**官方未对一般 ISV 开放**——「发布」官方路径只有分享 SDK handoff（用户确认）；市面宣称「API 支持定时/批量发笔记」的多为第三方爬虫/RPA 封装，非官方授权通道。微信视频号「视频内容发布 API」官方多次答复**暂未开放**，现有视频号 API 集中在电商/橱窗/直播数据，无视频上传发布写接口。故视频号侧「自动发布」连灰区都进不去，只有 Tier0 微信分享交接可走——**而该交接通道本身（G6 视频号侧）能否预填发布器尚存疑**。
>
> **为何抖音 video-create 是灰区而非绿区（修订核心论证）**：它同时满足灰区三判据——(1) 需新增服务端（破静态站不变量）；(2) 官方授权 + 仅单号；(3) **机器代操作账号、删除逐帖用户确认动作**。它与 RPA 黑区的唯一区别是「走官方授权通道、合规」，与绿区的根本区别是「人不在环、无逐帖手动 gate」。把它标绿会同时违反本草稿自定义判据与「无人确认的纯自动发布即便走官方 API 在多数平台仍是灰区」的审查铁律。

### 6.6 黑区 🔴：永久焊死 + 为什么（合规红线 + 技术形态双拦）

| 🔴 黑区项 | 合规红线 | 技术形态 |
|------|------|------|
| X1 内容采集（爬虫批量采集他人爆款图文/视频） | 小红书禁搬运（3 次限流/5 次封号）+ 抖音禁自动化程序接入收集信息；著作权侵权 | 需爬虫/RPA 运行时持续抓取，静态站 SOP 物理做不到，采集本身无创作价值 |
| X2 智能去重 / 查重规避（像素级二创规避原创度识别） | 目的即对抗平台查重风控掩盖搬运，助长禁止的搬运与虚假内容 | 依附采集+RPA 管线，价值导向违规，AICrew 永不承接「规避识别」类能力 |
| X5 多账号管理（指纹浏览器批量矩阵号） | 小红书禁恶意批量注册；微信把第三方工具/外挂接入（含多开/群控/协议号）列为违规 | 指纹浏览器+多线程对抗多账号风控，桌面 RPA 运行时，静态站架构外 |
| X6 智能养号（关键词自动化养号、机器涨粉） | 小红书 2025 封禁黑灰产超 1200 万、养号涨粉为治理核心；抖音禁养号买卖账号；微信禁刷粉 | 需 7×24 自动化行为模拟，静态站无常驻进程做不到且触红线 |
| X8 智能互动（自动评论/点赞/关注/私信） | 小红书禁诱导/机器互动；抖音禁外挂自动化；微信禁外挂刷粉赞评论 | 需账号自动化操作运行时，AICrew 既定边界=只产内容不操作账号 |
| X9 指纹浏览器（多账号风控对抗底座） | 存在目的即对抗风控、支撑批量违规，微信/抖音外挂条款直接覆盖 | 独立桌面客户端+浏览器内核改造，是「另一类软件形态」，与 Next.js 静态站根本不兼容 |
| X7「批量跨号定时、无人确认的 RPA 群发」半 | 小红书禁批量/高频/机器发布；微信/抖音外挂条款禁自动化批量操作 | 跨号无人值守群发需 RPA+指纹浏览器编排，静态站做不到（单号官方授权 API 服务端代发另算，见 6.5 灰区——但仍不是绿区） |
| X10「实时抓他人/全站数据」半 | 第三方爬虫式实时抓取属违规 | 跨源不可读+无常驻轮询面双重落空（官方授权读自有数据另算，见 6.5 灰区） |

> 黑区写入项目记忆，永久有效：素材采集/爬取、批量定时自动发布（RPA 跨号群发）、自动评论/点赞/关注、自动私信、指纹浏览器多账号 RPA、实时数据抓取（他人/全站）、像素级查重规避。这些**不是缺陷，是定位**。

### 6.7 一句话重构结论

RPA 自动化层**不是「铁板一块的禁区」，是一条「黑 🔴 → 灰 🟡 → 绿 🟢 的边界谱系」**，三段的唯一决定性分界是**「逐帖人工确认 gate 是否保留、人是否始终在环」**（官方授权只洗白合规、不能把能力抬进绿区）：

- 🟢 **绿区 = 内容产物半 + 人在环的官方交接**——内容产物半（X3 改写/X4 生成/一键克隆参考改写半，已强覆盖）、官方分享交接（小红书 Tier0 已落地+已验证、Tier0.5 基建齐备可做、Tier1 待凭证；抖音「分享到抖音」compose 交接为 `inferred` 待证实、视频号侧通道存疑）。**绿区不含任何服务端机器代发**；
- 🟡 **灰区 = 官方授权的机器代发 / 读自有数据**——单号 + 官方授权 + 需突破静态站加服务端、**且无逐帖人工确认 gate**。代表项即**抖音 upload-video + video-create 服务端单号发布**（本次从绿区移入此处）、官方 API 读自有账号数据、小红书 Tier1（含凭证）、微信小店电商 API；做成高频/跨号/抓他人即跌回黑区；
- 🔴 **黑区永久焊死**——爬虫采集 / 查重规避 / 机器养号 / 自动互动 / 指纹多开 / 实时抓他人数据 / RPA 跨号群发，被技术形态（静态站物理做不到）+ 2025 合规红线（小红书/抖音/微信外挂与机器行为条款）**双重不可逾越**地钉死。

绿区与灰区的分界线只有一条：**逐帖人工确认 gate 是否保留、人是否始终在环**（绿区保留、灰区删除）；灰区与黑区的分界线是：**是否走官方授权通道、是否仅限单号自有数据**（灰区走、黑区不走）。守住「人在环」这条线，「发布最后一公里」既补齐又不踩红线——这就是 AICrew 相对小鸡AI 矩阵系统的差异化护城河。
